# AI Coding Agent 正在进入开源供应链：180M 仓库普查说明了什么

## ⭐ Tier 0 Special Attention

今天选这篇不是因为它来自某个大厂发布会，而是因为它给 AI Coding 领域补上了一块更基础的测量底座：我们到底如何在真实开源生态里识别 AI coding agents 的足迹？过去很多讨论依赖问卷、个别平台数据、PR bot 账号或单一配置文件；这篇论文把问题推进到一个更可复用的方法论层面，用 World of Code 的 180M+ Git 仓库、四类检测信号、手工验证样本和跨渠道对比，说明“AI 写了多少代码”这件事本身很难被单一指标回答。

对 AI Native Engineering 来说，这属于 Tier 0 级别的专题材料：它不是又一个工具体验帖，而是提供了一套观测 AI agent adoption 的框架。后续我们看 Codex、Claude Code、Cursor、Copilot、OpenHands、Aider 等工具的采用率、质量、风险和组织影响时，都需要先回答一个问题：我们看到的是完整现实，还是某个渠道留下的影子？

## 核心洞察

这篇论文的核心价值，是把 AI coding agents 从“工具使用感受”拉回到“可观测、可审计、可复现的工程事实”。作者发现，AI agents 在开源供应链里的痕迹高度分散：有的通过 bot 账号发 PR，有的把签名写进 commit message，有的挂在人的作者名后面，有的只留下 `CLAUDE.md`、`AGENTS.md`、`.cursorrules` 这类配置文件。只看其中一种信号，就会系统性低估 adoption。

最关键的判断有五个。第一，AI coding agent adoption 不是一个单指标问题，PR、commit、配置文件和作者身份看到的是不同世界。第二，Claude Code 的 commit 足迹如果只看 bot 账号，会被低估约 30 倍。第三，到 V2604 快照时，commit-attributed agents 已经达到每月 320K+ commits 的规模。第四，PR 部署的 cloud agents 更容易表现为 feature work，commit/IDE 部署的 agents 更容易表现为 maintenance work，因此“agent 在做什么”很大程度上由部署模式决定。第五，AI adoption 的时序呈现两极化：一类项目从第一天就带着 AI 配置诞生，另一类成熟代码库在多年后接入 AI。

## 重点内容

### 1. 为什么 AI coding agent 的检测本身很难

论文从一个非常实际的问题开始：如果我们想研究 AI 对开发效率、代码质量和开源供应链风险的影响，首先要知道哪些代码、commit、PR 或项目受到了 AI agent 的参与。但这个前提并不简单。

传统 bot 比较容易识别，因为它们通常使用固定账号、固定消息模板、固定触发方式。AI coding agents 则不同：它们可能在本地 IDE 或 CLI 中运行，以人类开发者身份提交代码；也可能通过云端服务发起 PR；还可能只把配置文件写进仓库，不在每个 commit 中显式署名。作者强调，如果依赖单一信号，例如只看 bot account 或只看 PR，就会把大量“静默使用”排除在外。

这也是这篇论文相对重要的地方：它不直接问“AI 是否提高了生产率”，而是先解决“如何知道 AI 在哪里参与了开发”。对任何后续 productivity、quality、security 研究来说，这都是前置测量问题。

### 2. 四类痕迹：从 Bot 账号到静默配置文件

作者提出了一个四类检测 taxonomy。

**Type A 是 centralized bot account。** 这类 agent 用一个注册的 bot 身份提交代码，检测方式相对直接，可以通过固定 name / email 做 lookup。OpenHands 和 CodeRabbit 是这类例子。

**Type B 是 commit-message signature。** 这类 agent 在 commit message 中留下明确签名，例如 “Generated by Replit” 或 “Generated with Claude Code”。检测方式是对 commit message 做正则匹配。

**Type C 是 distributed human attribution。** 这类 agent 不使用中心化 bot，而是让不同开发者在作者名或提交信息中带上工具特定后缀。Aider 是典型例子。如果不扫描作者名模式，这类使用很容易被漏掉。

**Type D 是 configuration-file only，也就是 silent adoption。** 这类工具在 commit 层面几乎不留可识别痕迹，但仓库里会出现配置文件，例如 `copilot-instructions.md`、`CLAUDE.md`、`.claude/`、`.cursorrules`、`AGENTS.md` 等。它们不能直接证明每个 commit 都由 AI 生成，但可以作为项目级 adoption 的重要代理信号。

这套 taxonomy 的意义在于：AI agent 的“存在方式”已经超出了传统 bot 检测模型。未来企业内部如果要做 AI coding governance，也不能只靠一个字段、一个署名或一个平台导出的报表。

### 3. 180M+ 仓库普查看到的关键数字

论文使用 World of Code 的三个快照，覆盖 2024 年 12 月到 2026 年 4 月，合计 180M+ Git repositories。作者在这些快照上检测 12 类 agents，并对检测网格做了 495 个手工验证标签。

最有冲击力的数字是 Claude Code 的低估问题。在 V2510 快照中，如果只用 bot-account lookup，能找到 28,154 个 Claude Code commits；但加入 message signature 等多方法后，union 达到 850,157 个 commits。换句话说，单看 bot 账号会产生约 30 倍的相对召回差距。

到 V2604 快照，commit-attributed agents 已经达到每月 320K+ commits 的规模。Claude Code 以 886,122 commits、17,295 projects 领先；同时，在 Type D 配置文件层面，Claude 也成为 dominant silent agent，出现在 21,078 个 projects 中，并有 888,177 个 blob occurrences。

这里要注意单位差异。commit count、project count、blob occurrence 不是同一个指标，不能简单横向排名。commit count 更接近“可归因的提交活动”，project count 更接近“项目级采用”，blob occurrence 还受到文件反复修改的影响。真正重要的是：不同指标揭示的是 adoption 的不同侧面。

### 4. PR 视角和 Commit 视角几乎看到两个世界

论文把自己的 commit-based census 和 AIDev 这个 PR-based census 做了对比。AIDev 覆盖 932,791 个 agent-authored PRs，来自 Codex、Devin、GitHub Copilot、Cursor 和 Claude Code。对比结果显示，两个渠道捕捉到的人群和工作类型高度不重合。

PR 渠道里，Codex 是最大的 agent；但在 commit 渠道里，Codex 几乎不可见。反过来，Claude Code 在 commit 渠道里非常显著，但在 PR 渠道里接近不可见。论文指出，PR-based census 会漏掉 79% 的 commit-detected Claude Code adopters，而 commit-based census 基本会漏掉 Codex adopters。

更有意思的是任务类型差异。PR-deployed cloud agents，例如 Codex 和 Cursor，更容易在数据中呈现为 feature work；commit-deployed 或 editor-deployed agents，例如 Claude Code、OpenHands 和 Aider，更容易呈现为 maintenance work。也就是说，我们看到的“agent 擅长做什么”，不一定是模型能力本身决定的，也可能是部署入口、权限边界和协作界面决定的。

### 5. Adoption 时间呈两极化：Born-with-AI 与 Legacy Integration

作者还研究了 adoption timing。以 GitHub Copilot 配置项目样本为例，采用时间呈现明显的 bimodal pattern：一类项目在创建后的 0 到 6 天内就加入 AI 配置，属于 born-with-AI；另一类项目则是在多年甚至十几年后，把 AI 配置引入成熟代码库。

OpenHands 的样本也体现了类似倾向：在成功检索的 28 个 adopting projects 中，19 个项目，也就是 68%，在 30 天内采用 OpenHands；其余 32% 则是 5 到 1,516 天后采用。作者据此估计，OpenHands adoption 更集中在 greenfield / near-greenfield 项目中。

这个发现对工程管理很有价值。AI 工具对新项目和老项目的影响机制很可能不同。新项目里，agent 可能直接影响架构、目录结构、测试策略和初始工程习惯；老项目里，agent 更多是在既有约束下做维护、修复、迁移和增量开发。把这两类项目混在一起计算生产率，很容易得出误导性结论。

### 6. 速度、质量和风险：论文保持了必要克制

论文也做了 velocity 和 quality 相关的描述性分析，但作者很清楚地强调，这些不是严格因果估计。比如 OpenHands adoption 的 naive before-after analysis 显示，非 fork adopters 几乎都出现 post-adoption commit velocity 增长；greenfield / near-greenfield 项目的 median increase 很高，legacy projects 也有明显增长。但这些数字会受到快照时间、项目选择、摄入延迟等影响，因此只能说明 adoption 与活动变化同时出现，不能直接等同于 AI 导致生产率提升。

在质量代理指标上，作者没有发现粗粒度的 velocity-quality tradeoff。Claude Code、OpenHands 和 Aider 的 AI commits 在样本中表现出更高的 bug-fix commit rates，并且 temporal SZZ ratio 低于 human baseline。不过作者也提醒，bug-fix keyword、revert rate、SZZ adjacency 都是间接指标，需要更细粒度的 diff-level analysis 验证。

这份克制很重要。AI coding 领域现在最容易出现的误区，是把 adoption、activity、velocity、quality、business value 混成一个故事。论文的态度更健康：先把观测通道和偏差讲清楚，再谈影响。

### 7. 数据与复现：走向标准化 AI Attribution

论文最后提到 replication package 会包含 Type A–D 的检测脚本、ClickHouse / shard scan drivers、canonical agent-pattern file、495-label validation set、AIDev comparison code，以及每个数字到脚本和输出的映射。未来工作包括扩展到 V2605 快照、覆盖更多 agents，例如 Gemini Code Assist、OpenAI Codex / ChatGPT beyond AGENTS.md，并发展标准化的 AI attribution metadata for Git commits。

这也是最值得组织借鉴的一点。今天很多 AI coding adoption 还是靠工具后台、主观汇报或开发者自觉署名。长期看，工程组织需要更标准化的 attribution metadata：哪个 agent 参与了哪个 change？人类审查到什么程度？测试和验证证据在哪里？哪些改动是 AI suggested、AI authored、human modified，哪些只是 AI-assisted？没有这些元数据，后续的质量、合规和效率分析都会不稳。

## 给我们的启示

对 AI Native Engineering 来说，这篇论文至少带来五个直接启示。

第一，内部度量不能只看一个平台的报表。Codex、Claude Code、Cursor、Copilot、Aider、OpenHands 的部署入口不同，留下的痕迹也不同。组织要理解 AI coding adoption，需要同时看 PR、commit、配置文件、agent metadata、CI logs 和 review records。

第二，AI attribution 应该成为工程系统的一等元数据。我们可以要求所有 agent-generated 或 agent-assisted changes 都通过统一 trailer、PR label、branch naming、commit note 或 metadata file 记录来源，并把 human review、test evidence、rollback path 也纳入同一条链路。

第三，greenfield 和 legacy 项目要分开管理。新项目应该提供 AI-ready scaffold，包括测试、CI、lint、security policy、AGENTS.md、权限边界和验收模板；老项目则需要更强的 context packaging、change-risk classification、incremental rollout 和 human owner。

第四，不要把“AI commit 增多”直接等同于生产率提升。更合理的指标组合应该包括 lead time、review effort、change failure rate、revert rate、escaped defects、test coverage delta、incident link、developer cognitive load，以及 AI 生成内容最终存活率。

第五，AI coding governance 的重点不是阻止使用，而是让使用可见。只要 agent 能写代码、跑命令、改文件、发 PR，它就已经进入供应链。真正的风险不是 AI 参与，而是 AI 参与但没有被记录、没有边界、没有审查、没有证据。
